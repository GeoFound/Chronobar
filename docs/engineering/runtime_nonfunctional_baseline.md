# Chronobar 运行时非功能基线 v1.0

## 1. 文档定位

本文档定义 Chronobar 在正式进入 P1 / M2 实现前必须冻结的非功能约束，包括性能目标、资源边界、基准测试方法、最小可观测性要求与健康模型。

它不替代协议文档，也不承诺已经存在对应运行时代码；它的作用是为后续实现提供统一的验收标尺，避免在热路径选型、数据链路与桌面侧恢复策略上反复返工。

## 2. 总体原则

- **先定义预算，再写热路径**：任何 Tick / Event / Bar 热路径实现都必须先对齐吞吐、延迟和内存预算。
- **先定义测法，再宣称性能**：未经基准脚本或可复现实验验证，不得在文档中写入“高性能”“低延迟”“可大规模处理”等模糊表述。
- **性能与正确性并重**：性能优化不得破坏 trading_date、session、回放一致性与桥接边界。
- **桌面优先**：所有预算默认面向单机桌面运行，不以服务端集群假设掩盖问题。

## 3. 正式范围与预算前提

本基线建立在以下前提上：

- 单机、单用户、单 Python sidecar
- 单账户优先，单活跃策略运行时
- 一条规范数据主链路：`Gateway/Replay -> Event -> Rule -> Compute -> Storage -> UiBridge`
- 历史 Tick 使用 `DuckDB + Parquet`，元数据使用 `SQLite`

若后续突破这些前提，必须先修订本文件。

## 4. 性能目标

### 4.1 P1 / M2 最小目标

P1 / M2 的目标不是追求机构级极限性能，而是建立**可测量、可解释、可持续优化**的单机基线。

最低目标如下：

- **Tick 输入吞吐**：单合约标准 Tick 输入链路目标 **>= 10,000 ticks/s**
- **Tick -> Event -> Rule -> Bar 热路径延迟**：在无 UI 推送压力下，P95 **< 20ms**
- **Parquet 批量落盘**：单次 flush 不得造成持续性事件堆积；目标为每次 flush 后队列可在 **5 秒内** 回到健康水位
- **DuckDB 本地查询**：单日、单合约 Tick 查询应在开发机上保持交互式速度，目标 **< 1 秒** 完成常见过滤查询
- **回放性能目标（P3 / M4 预留）**：1 个合约、1 年本地历史数据的最小回测链路目标 **< 5 分钟** 完成

说明：

- 上述值是实现基线，不是最终上限。
- 若运行环境低于开发基线，可接受降级，但必须在 benchmark 结果中说明硬件条件。

### 4.2 资源预算

- **Event queue 默认容量**：沿用协议中的 `10000`
- **高水位告警阈值**：队列深度达到 `8000` 必须产生日志与健康告警
- **单次历史数据加载内存上限**：沿用回测协议的 **1GB** 上限
- **长期运行内存预算**：M2 最小闭环在常规桌面环境下不得出现无界内存增长

## 5. 基准测试要求

进入正式实现后，至少建立以下基准：

### 5.1 必备 benchmark 套件

- **tick_ingest_benchmark**：测量标准 Tick 从输入到事件入队的吞吐
- **bar_aggregation_benchmark**：测量 Tick -> Bar 的聚合成本
- **storage_flush_benchmark**：测量 TickCache -> Parquet flush 吞吐与阻塞时间
- **duckdb_query_benchmark**：测量常见筛选、时间范围查询与回放读取速度
- **replay_consistency_benchmark**：测量回放链路吞吐，并与实时链路做一致性比对

### 5.2 基准记录要求

每次记录 benchmark 结果时，必须同时记录：

- 代码版本 / commit
- Python 版本
- 操作系统
- CPU / 内存基础信息
- 输入数据规模（合约数、Tick 数、时间范围）
- 是否包含 UI Bridge 推送

未经上述元数据标注的性能结果，不得作为阶段验收证据。

## 6. 热路径技术分工

为减少选型漂移，Chronobar 的正式建议如下：

- **Pydantic**：用于配置对象、迁移、业务校验和用户输入边界
- **msgspec**：用于高频对象编解码、事件日志、回放数据快路径
- **dataclass / Enum / Protocol**：用于协议对象、抽象边界与可读性优先的核心模型
- **structlog**：用于结构化日志、trace_id 传播、审计与 JSON 输出

约束：

- 不让 `Pydantic` 成为 Tick 热路径的默认对象容器
- 不让 `msgspec` 反向承担复杂 migration / config 业务逻辑
- 不因优化而引入第二套长期并存的数据口径

## 7. 最小可观测性基线

### 7.1 日志

正式运行时至少应输出以下结构化字段：

- `trace_id`
- `event_type`
- `source`
- `instrument_id`（如适用）
- `mode`（live / sim / replay）
- `latency_ms`（如适用）
- `queue_depth`（如适用）
- `status`

### 7.2 指标

最小指标面包括：

- 事件队列深度
- Tick 落盘延迟
- Tick flush 失败次数
- UI Bridge 订阅连接数
- 回放推进速度
- sidecar 健康状态

### 7.3 健康模型

至少定义以下健康状态：

- `healthy`：关键链路在预算内
- `degraded`：功能可用，但存在延迟、积压、缺失降级或恢复动作
- `unhealthy`：关键链路不可用，需要用户或系统执行恢复

## 8. 告警与降级策略

当出现以下情况时，必须进入降级或告警路径：

- Event queue 高水位持续超过阈值
- Tick flush 连续失败
- DuckDB 查询不可用
- 回放结果与实时链路差异超出容忍范围
- sidecar 健康检查失败

最低降级动作包括：

- 对低优先级 UI 推送做节流或批量化
- 暂停非关键订阅
- 保留日志与诊断快照
- 向前端暴露用户可见状态，而不是静默失败

## 9. 阶段性门禁

### 9.1 M1 / P0 必须完成

- 性能目标已成文
- benchmark 套件清单已成文
- 最小可观测性字段已成文
- 健康状态与高水位策略已成文

### 9.2 P1 / M2 必须验证

- 至少 1 组 Tick 吞吐 benchmark 可运行
- 至少 1 组 Bar 聚合 benchmark 可运行
- 日志字段与健康状态在最小闭环中可观察

### 9.3 P3 / M4 必须验证

- Replay / Backtest benchmark 可运行
- 1 年数据回测目标有可复验测量结果

## 10. 禁止事项

- 禁止只写“高性能”“低延迟”而不写预算与测法
- 禁止在无 benchmark 的前提下引入大规模热路径重构
- 禁止让 UI 体验需求反向破坏数据主链路语义
- 禁止把监控与恢复视为发布前装饰项
