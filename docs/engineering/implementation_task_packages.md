# Chronobar 实施任务包 v1.1

## 1. 文档定位

本文档把 [`delivery_master_plan.md`](delivery_master_plan.md) 拆成可交给任务 AI 或开发者逐个执行的任务包。

任务包的编排遵循以下主线：

- 先覆盖个人交易者完整核心工作流，再决定哪些能力不做
- 保持“全而小”，不把任务包演化成机构平台式功能铺开
- 默认优先本地、自有、低外部依赖的实现路径
- 先建立本地数据飞轮，再扩功能面
- 先验证中国期货语义正确性，再扩展策略复杂度
- 先形成可一键运行的内建样板，再考虑开放生态
- sidecar 生命周期治理与桌面故障恢复不属于发布前装饰项，而属于正式开发范围

每个任务包都必须有：

- 明确目标
- 依赖输入
- 产出文件或模块
- 验证方法
- 阻塞条件
- 进入下一任务的前提

## 2. 使用规则

- 一次只推进一个任务包到 `in_progress`
- 每个任务包结束前必须先做验证，再进入下一个任务包
- 若验证失败且涉及协议边界，应回退到上一阶段 gate，而不是硬继续写代码
- 若任务包只完成了功能实现、但没有补齐测试 / 文档 / 恢复 / 运维入口，则不得视为完成
- 阶段关闭包、已知限制、回滚说明属于正式产物，不得省略

## 3. 跨阶段任务维度

从当前仓库状态到最终产品化落地，所有阶段都必须覆盖以下任务维度；区别只在于每个阶段覆盖到什么深度。

### 3.1 任务维度清单

- **D1 协议与模型维度**：核心对象、协议映射、版本治理、兼容边界
- **D2 运行链路维度**：数据输入、事件流、计算、桥接、前端消费
- **D3 数据资产维度**：落盘、查询、回放、导出、备份、恢复、损坏处理
- **D4 正确性维度**：夜盘语义、回放一致性、撮合正确性、策略权限正确性
- **D5 非功能维度**：性能目标、benchmark、内存预算、延迟、容量、高水位治理
- **D6 可观测性维度**：日志、指标、诊断、健康状态、恢复入口
- **D7 体验与工作流维度**：工作区、回放、图表、风控、错误反馈、安装升级体验
- **D8 安全与治理维度**：权限矩阵、审计、AI 遏制、插件边界、发布回滚
- **D9 测试与验证维度**：单测、集成、契约、回归、端到端、安装验证、恢复演练
- **D10 文档与发布维度**：README、样例、runbook、已知限制、发布说明、beta/GA 清单

### 3.2 阶段覆盖规则

- P0 必须先把 D1 / D3 / D5 / D10 写清楚，避免后续返工
- P1-P4 必须逐步把 D2 / D4 / D6 / D9 做成可验证事实，而不是停留在文档
- P5 必须在 D8 / D9 / D10 上显著加强，防止 AI 成为越权入口
- P6 必须把所有维度统一收口到 beta / GA 放行机制中

### 3.3 每阶段关闭时必须沉淀的产物

每个阶段结束时，至少要沉淀：

- 已入仓的代码或文档产物
- 可重复运行的验证命令
- 对应样例 / 基准 / 契约 / 诊断材料
- 已知限制与 blocker 清单
- 回滚动作与降级说明
- 是否允许进入下一阶段的明确结论

## 4. P0：M1 收口任务包

### P0-T1 当前阶段与真相源统一

- **目标**：统一 README、roadmap、AI 指南、贡献入口与测试断言中的当前阶段口径
- **依赖**：M1 协议与配置基线已存在
- **产出**：阶段真相源文档、统一后的导航表述、与之对齐的仓库基线测试
- **验证**：仓库内不再出现“当前仍是 M1”与“P1 已完成”的正式口径冲突
- **阻塞条件**：团队对当前正式阶段仍存在分歧

### P0-T2 运行范围冻结

- **目标**：在进入正式实现前冻结单机、单用户、单账户、单活跃策略等早期边界
- **依赖**：P0-T1
- **产出**：范围冻结说明、阶段边界说明、对后续计划文档的引用链
- **验证**：执行文档与贡献入口均引用相同的范围边界
- **阻塞条件**：需要提前承诺多账户或多策略并发正式支持

### P0-T3 存储生命周期与恢复基线

- **目标**：为 Tick/Bar/配置/工作区建立统一的数据生命周期、备份/恢复与损坏处理约束
- **依赖**：P0-T2
- **产出**：存储目录约定、保留策略、恢复模型、用户可见恢复入口说明
- **验证**：系统架构、工程基线和交付计划均能引用同一恢复设计锚点
- **阻塞条件**：存储主路径仍存在多套相互竞争的数据口径

### P0-T4 非功能基线与 benchmark 约束

- **目标**：在正式实现前冻结性能目标、资源预算、可观测性要求与 benchmark 套件清单
- **依赖**：P0-T2
- **产出**：非功能基线文档、最小性能目标、健康状态定义、benchmark 清单
- **验证**：工程基线与后续任务包已明确引用非功能基线
- **阻塞条件**：性能目标无法与桌面单机约束达成一致

### P0-T5 M1 gate 复核

- **目标**：在新增基础设计文档后重新执行一次 M1 gate，确保进入实现前的地基闭合
- **依赖**：P0-T1 ~ P0-T4
- **产出**：更新后的 README / roadmap / baseline / tests / changelog
- **验证**：`just docs-check` 通过；若本地依赖可用，`just check` 作为进入 P1 前门禁
- **进入下一阶段前提**：P0 gate 全部通过

## 5. P1：M2 核心闭环任务包

### P1-T1 仓库运行时代码初始化

- **目标**：创建最小 Python runtime 目录骨架
- **依赖**：P0-T5
- **产出**：`app/`、`core/`、`rules/`、`compute/`、`storage/`、`ui/desktop/`、`tests/unit/`、`tests/integration/`
- **验证**：目录结构与 `engineering_baseline.md` 一致；`pytest` 可发现新增测试目录
- **阻塞条件**：目录设计与现有协议边界冲突

### P1-T2 标准对象与异常层

- **目标**：把协议中的最小对象映射到运行时代码
- **依赖**：P1-T1
- **产出**：核心模型、枚举、异常定义
- **验证**：对象可实例化；协议测试通过；mypy 不报公共接口错误
- **阻塞条件**：需要改协议字段才能实现

### P1-T3 EventEngine

- **目标**：实现统一事件总线
- **依赖**：P1-T2
- **产出**：事件注册、订阅、分发、trace_id 保留、异常隔离
- **验证**：事件投递单测与集成测试通过
- **阻塞条件**：事件模型与 `event_protocol.md` 不一致

### P1-T4 SimGateway 与本地 Tick 输入

- **目标**：建立无外部网络依赖的第一数据入口，并验证 Gateway Adapter 的最小能力声明路径，避免 M2 被真实 Gateway 阻塞
- **依赖**：P1-T2、P1-T3
- **产出**：`SimGateway` 或等价本地回放输入模块；最小 Tick 样例数据；启动/停止/订阅接口；最小能力声明样例
- **验证**：可稳定回放至少 1 个合约的 Tick 样例，并进入事件总线；上层可读取该 Gateway 的能力范围与降级边界
- **阻塞条件**：输入接口与 `gateway_protocol.md` 或事件模型不一致

### P1-T5 Tick 持续落盘与 DuckDB 查询入口

- **目标**：建立最小本地数据飞轮起点
- **依赖**：P1-T4
- **产出**：TickCache、Parquet 落盘、DuckDB 查询入口、最小目录约定
- **验证**：回放 Tick 后可在本地查到同口径数据；读写路径可自动化测试
- **阻塞条件**：落盘数据结构与协议字段不一致，或查询结果不可复验

### P1-T6 RuleEngine 最小 session 判定

- **目标**：实现夜盘与 trading_date 最小规则
- **依赖**：P1-T2、P1-T4
- **产出**：交易时段模板加载、session 判定
- **验证**：夜盘样例测试通过；交易日归属与 session 切分可复验
- **阻塞条件**：规则样例与协议描述冲突

### P1-T7 BarAggregator + 1 个指标

- **目标**：完成 Tick -> Bar 与最小指标计算，并验证 Bar 归属不破坏交易日语义
- **依赖**：P1-T3、P1-T6
- **产出**：Bar 聚合器、至少 1 个指标
- **验证**：1 分钟 Bar 聚合测试通过；夜盘 Bar 的 session / trading_date 一致性测试通过
- **阻塞条件**：计算结果不稳定或不可复验

### P1-T8 UiBridge 最小投影

- **目标**：向前端提供最小 Query / Subscription 投影
- **依赖**：P1-T3、P1-T7
- **产出**：最小桥接接口与标准返回结构
- **验证**：契约测试与样例结构一致
- **阻塞条件**：桥接字段与协议样例偏离

### P1-T9 核心闭环演示

- **目标**：把前面任务串成可运行闭环
- **依赖**：P1-T1 ~ P1-T8
- **产出**：开发演示脚本 / README / 集成测试
- **验证**：SimGateway -> Tick 落盘 -> DuckDB 查询 -> Event -> Bar -> Indicator -> Snapshot 全链路可复现

### P1-T10 基准烟囱与健康诊断

- **目标**：让 P1 不只是“能跑”，而是具备最小性能观测和健康诊断能力
- **依赖**：P1-T3、P1-T5、P1-T7、P1-T8
- **产出**：tick ingest benchmark、bar aggregation benchmark、队列深度与落盘延迟诊断输出
- **验证**：至少 1 组基准结果可复现；高水位时可输出诊断信息

### P1-T11 存储恢复烟囱

- **目标**：验证最小数据目录、导出与损坏处理路径不是纸面设计
- **依赖**：P1-T5
- **产出**：最小数据目录实现、导出样例、损坏分片隔离或等价恢复路径
- **验证**：最小恢复路径可演示；数据损坏不会导致系统静默崩溃

### P1-T12 P1 阶段关闭包

- **目标**：形成进入 P2 前的正式阶段通过材料
- **依赖**：P1-T1 ~ P1-T11
- **产出**：阶段总结、benchmark 结果、已知限制、回滚点、下一阶段输入清单
- **验证**：P1 gate、统一 gate 模板与阶段证明材料全部满足
- **进入下一阶段前提**：P1 gate 全部通过

## 6. P2：M3 插件系统任务包

### P2-T1 Manifest 与注册表

- **目标**：定义插件发现、注册、版本与依赖的最小治理面
- **依赖**：P1-T12
- **产出**：manifest schema、注册表结构、builtin 与第三方插件命名约定
- **验证**：合法/非法 manifest 样例可区分；依赖缺失可报明确错误

### P2-T2 PluginManager

- **目标**：实现插件加载、禁用、卸载与生命周期管理
- **依赖**：P2-T1
- **产出**：PluginManager、插件状态机、加载顺序控制
- **验证**：插件可发现、加载、禁用、卸载；重复加载与缺失插件有确定行为

### P2-T3 Host API 最小实现

- **目标**：提供只覆盖当前闭环的最小 Host API，而不是提前铺满全部能力
- **依赖**：P2-T2
- **产出**：读取市场数据、读取指标结果、输出信号、受控命令入口
- **验证**：插件可通过 Host API 完成一个完整的内建样板链路

### P2-T4 权限矩阵与隔离

- **目标**：实现 M3 默认隔离级别：线程级隔离 + 权限门禁 + 异常隔离
- **依赖**：P2-T2、P2-T3
- **产出**：权限检查、插件异常捕获、隔离执行边界
- **验证**：异常插件不会拖垮主流程；未授权调用被拒绝

### P2-T5 Builtin 插件样例

- **目标**：形成一条可以一键运行的内建插件样板，而不是先开放生态
- **依赖**：P2-T3、P2-T4
- **产出**：至少 2 个内建指标/信号插件、样板脚本、样板测试
- **验证**：在无外部网络条件下可运行完整插件闭环

### P2-T6 插件卸载 / 缺失恢复

- **目标**：保证工作区或运行链路在插件缺失时可降级恢复
- **依赖**：P2-T2、P2-T5
- **产出**：卸载策略、缺失插件恢复逻辑、降级说明
- **验证**：卸载或缺失插件后系统仍可进入稳定降级状态

### P2-T7 插件权限矩阵测试包

- **目标**：把权限边界做成自动化验证，而不是靠人工约定
- **依赖**：P2-T4
- **产出**：权限矩阵样例、允许/拒绝测试集、异常插件样例
- **验证**：插件越权访问、越权命令、越权网络访问均能被拒绝或审计

### P2-T8 工作区与插件配置迁移

- **目标**：保证插件版本变化不会直接破坏用户工作区、插件配置与相关策略引用这些用户资产
- **依赖**：P2-T1、P2-T6
- **产出**：插件配置迁移策略、工作区缺失插件降级规则、策略引用兼容规则、兼容性样例
- **验证**：升级 / 卸载 / 缺失场景下工作区仍可恢复；用户资产不因插件变化而静默失效

### P2-T9 P2 阶段关闭包

- **目标**：形成进入回测阶段前的插件治理通过材料
- **依赖**：P2-T1 ~ P2-T8
- **产出**：插件样板、权限测试、降级说明、阶段总结、回滚点
- **验证**：P2 gate、统一 gate 模板与阶段证明材料全部满足

统一验证要求：
- 插件加载路径可测试
- 未授权调用可拒绝
- 插件异常不拖垮主流程
- 工作区对缺失插件具备降级能力
- 内建样板可一键运行

## 7. P3：M4 回测系统任务包

### P3-T1 HistoricalDataLoader

- **目标**：优先消费本地 Parquet 落盘数据，避免再引入第二套早期数据口径
- **依赖**：P2-T9
- **产出**：HistoricalDataLoader、本地分片读取、最小筛选接口
- **验证**：可从本地落盘数据加载指定合约/时间段数据

### P3-T2 Replay Engine

- **目标**：建立与实时链路兼容的回放执行入口
- **依赖**：P3-T1
- **产出**：Replay Engine、回放控制接口、时间推进机制
- **验证**：回放数据可重放为标准事件流

### P3-T3 BacktestEngine

- **目标**：以共享协议对象与共享主链路逻辑为原则实现回测引擎
- **依赖**：P3-T2
- **产出**：BacktestEngine、最小策略执行循环
- **验证**：最小策略可运行，且不复制一套独立数据对象体系

### P3-T4 OrderMatcher

- **目标**：提供最小可解释撮合，不提前做过度复杂的市场微结构模拟
- **依赖**：P3-T3
- **产出**：限价单撮合、成交结果对象、最小费用/滑点钩子
- **验证**：撮合结果可通过手工样例复核

### P3-T5 Result / Report Builder

- **目标**：形成可复验的回测与研究输出，而不是先追求复杂报告样式
- **依赖**：P3-T3、P3-T4
- **产出**：收益、回撤、胜率、交易明细、因子评估摘要与最小实验导出能力
- **验证**：输出结果稳定且可重复生成；至少一条研究配置可复跑

### P3-T6 回放一致性测试

- **目标**：验证回放与实时链路共享同一口径，而不是只是“都能跑”
- **依赖**：P3-T1 ~ P3-T5
- **产出**：一致性测试集、黄金样例、差异定位输出
- **验证**：关键场景下一致性可自动化检查

### P3-T7 历史数据 benchmark 与黄金数据集

- **目标**：把回测阶段的性能与正确性固定到可重复样例上
- **依赖**：P3-T1、P3-T2、P3-T6
- **产出**：黄金数据集、回放 benchmark、1 年数据回测基准记录格式
- **验证**：关键 benchmark 能稳定运行；结果带硬件与数据规模说明

### P3-T8 回测导出与诊断包

- **目标**：让回测结果具备复核、归档和问题定位能力
- **依赖**：P3-T5、P3-T6
- **产出**：回测报告导出、交易明细导出、诊断包或等价差异报告
- **验证**：输出可复核，可作为 beta 前内部审查材料

### P3-T9 P3 阶段关闭包

- **目标**：形成进入桌面产品阶段前的统一通过材料
- **依赖**：P3-T1 ~ P3-T8
- **产出**：一致性证明、benchmark 结果、导出样例、回滚点、阶段总结
- **验证**：P3 gate、统一 gate 模板与阶段证明材料全部满足

统一验证要求：
- 回测策略样例可运行
- 输出结果可复验
- 与实时链路共享协议对象
- 与本地落盘 / 回放链路共享统一数据口径
- 至少一条研究配置或因子实验可复跑并可归档

## 8. P4：M5 产品壳任务包

### P4-T1 Tauri + React 初始化

- **目标**：建立桌面工作站壳，而不是只做静态前端原型
- **依赖**：P3-T9
- **产出**：Tauri 工程、React + TypeScript 初始化、基础运行脚本
- **验证**：可稳定拉起桌面壳并接入最小前端入口

### P4-T2 工作区壳与导航

## 9. P5：AI Assistant 任务包

### P5-T1 Session State 与记忆边界

- **目标**：建立 AI 会话态、工作区记忆与用户记忆的受控边界，避免 AI 直接读取或污染核心运行状态
- **依赖**：P4-T11
- **产出**：Session State、记忆分层规则、上下文裁剪规则、可审计的状态注入边界
- **验证**：AI 会话重置、工作区切换、权限变更、记忆清理时状态行为明确且可测试

### P5-T2 Knowledge Retriever 与 External Context Gateway

- **目标**：让 AI 回答优先基于仓库文档、本地数据、本地研究资产与受控知识源，并在必要时受控访问真实世界外部来源
- **依赖**：P5-T1
- **产出**：知识检索入口、外部上下文网关、研究资产检索入口、来源对象、知识源配置、白名单策略
- **验证**：回答可回溯到具体来源片段或研究资产；外部来源访问有可见授权与来源留痕

### P5-T3 Intent Router

- **目标**：把问答、解释、修改、执行请求分开治理
- **依赖**：P5-T1、P5-T2
- **产出**：Intent Router、风险等级分类、动作分级
- **验证**：高风险请求和低风险请求有不同执行路径

### P5-T4 Policy Engine

- **目标**：把拒答、确认点、越权限制做成硬约束
- **依赖**：P5-T3
- **产出**：策略规则、拒答模板、确认点机制、可配置治理项
- **验证**：越权请求、高风险操作、无证据回答均可被拒绝或拦截

### P5-T5 Tool Executor

- **目标**：提供受控工具执行，不允许 AI 直接绕过产品边界
- **依赖**：P5-T3、P5-T4
- **产出**：工具注册、权限校验、执行审计、失败回传结构
- **验证**：工具调用必须经过策略和权限检查；失败可追踪

### P5-T6 Candidate Validator / Change Set Builder

- **目标**：让修改类输出成为可应用的候选正式产物，而不是一次性文本草稿
- **依赖**：P5-T4、P5-T5
- **产出**：最小可用校验、change set 结构、影响范围摘要、待确认状态模型
- **验证**：候选配置、候选代码或候选工作区对象都能输出最小校验结果与待确认变更

### P5-T7 Evidence Binder 与 Answer Composer

- **目标**：让最终回答带来源、限定条件、不确定性与动作边界
- **依赖**：P5-T2、P5-T4、P5-T5、P5-T6
- **产出**：证据绑定、回答合成、来源展示、置信与限制展示
- **验证**：回答能够清晰区分“事实 / 推断 / 建议 / 未知”

### P5-T8 Audit Logger

- **目标**：保证 AI 输入、检索、工具调用、输出都可审计
- **依赖**：P5-T5、P5-T6、P5-T7
- **产出**：AI 审计日志、事件模型、导出接口
- **验证**：关键会话、记忆操作、外部访问和候选产物应用都可回放审计链路

### P5-T9 首批解释能力面

- **目标**：先把真正可控、可验证的 3 个产品能力做成，而不是泛化成“什么都能问”

- **产出**：产品问答、回测解读、风控解释
- **验证**：每个能力面都有样例、拒答样例、来源样例与评测样例

### P5-T10 用户资产构建与确认应用链路

- **目标**：把配置、工作区、指标或策略的候选产物生成与确认应用做成正式产品链路，并明确构建态/运行态边界
- **依赖**：P5-T4、P5-T6、P5-T7, P5-T8
- **产出**：候选产物生成、确认应用命令、拒绝命令、应用回执、回滚说明、构建态到运行态的发布边界说明
- **验证**：候选产物在确认前不会落正式对象；确认后可生成可追踪回执；未确认候选产物不会被运行态直接消费

### P5-T11 真实世界研究与记忆治理

- **验证**：用户可查看、关闭或删除外部访问与记忆相关状态；越权访问被拒绝

### P5-T12 AI 评测与红线测试

- **验证**：AI 评测可重复运行；核心红线场景通过；研究类回答不会跳过研究态到运行态发布边界

### P5-T13 P5 阶段关闭包

- **依赖**：P5-T1 ~ P5-T12

首批能力面：

统一验证要求：
- 来源显示存在
- 拒答策略生效
- 修改类动作需确认

## 11. 任务 AI 执行约束

交给任务 AI 执行时，必须遵守：

- 不得跨阶段抢跑
- 不得在 gate 未通过时开始下一阶段
- 不得用临时字段绕过协议
- 不得以“先写出来再整理”为理由污染边界
- 不得默认开启高风险 AI 能力
- 所有新增模块都必须附最小测试与验证命令
- 阶段关闭包、已知限制、回滚说明属于正式产物，不得省略

## 12. 任务包完成定义

一个任务包只有在以下条件全部满足时才算完成：

- 代码或文档产出已入仓
- 对应测试通过
- README / 契约 / 样例已同步（如适用）
- 风险与限制已记录
- 下一任务包所需输入已经稳定

## 13. 最终产品完成定义

从当前仓库状态到“真正产品化落地”，最终完成必须同时满足以下条件：

- P0-P6 全部阶段 gate 通过
- 安装、升级、回放、回测、桌面恢复、基础 AI 能力均存在明确可执行路径
- 数据资产、工作区、日志与诊断包可导出、可恢复、可审计
- beta blocker 全部关闭，GA 放行清单完成签收
- 仓库中不存在“代码能跑但没有测试 / 没有恢复 / 没有文档 / 没有回滚”的悬空阶段
