# Chronobar 实施任务包 v1.0

## 1. 文档定位

本文档把 [`delivery_master_plan.md`](delivery_master_plan.md) 拆成可交给任务 AI 或开发者逐个执行的任务包。

任务包的编排遵循以下主线：

- 先覆盖个人交易者完整核心工作流，再决定哪些能力不做
- 保持“全而小”，不把任务包演化成机构平台式功能铺开
- 默认优先本地、自有、低外部依赖的实现路径
- 先建立本地数据飞轮，再扩功能面
- 先验证中国期货语义正确性，再扩展策略复杂度
- 先形成可一键运行的内建样板，再考虑开放生态
- sidecar 生命周期治理与桌面故障恢复不属于发布前装饰项，而属于正式开发范围

每个任务包都必须有：

- 明确目标
- 依赖输入
- 产出文件或模块
- 验证方法
- 阻塞条件
- 进入下一任务的前提

## 2. 使用规则

- 一次只推进一个任务包到 `in_progress`
- 每个任务包结束前必须先做验证，再进入下一个任务包
- 若验证失败且涉及协议边界，应回退到上一阶段 gate，而不是硬继续写代码

## 3. P1：M2 核心闭环任务包

### P1-T1 仓库运行时代码初始化

- **目标**：创建最小 Python runtime 目录骨架
- **依赖**：M1 gate 完成
- **产出**：`app/`、`core/`、`rules/`、`compute/`、`storage/`、`ui/desktop/`、`tests/unit/`、`tests/integration/`
- **验证**：目录结构与 `engineering_baseline.md` 一致；`pytest` 可发现新增测试目录
- **阻塞条件**：目录设计与现有协议边界冲突

### P1-T2 标准对象与异常层

- **目标**：把协议中的最小对象映射到运行时代码
- **依赖**：P1-T1
- **产出**：核心模型、枚举、异常定义
- **验证**：对象可实例化；协议测试通过；mypy 不报公共接口错误
- **阻塞条件**：需要改协议字段才能实现

### P1-T3 EventEngine

- **目标**：实现统一事件总线
- **依赖**：P1-T2
- **产出**：事件注册、订阅、分发、trace_id 保留、异常隔离
- **验证**：事件投递单测与集成测试通过
- **阻塞条件**：事件模型与 `event_protocol.md` 不一致

### P1-T4 SimGateway 与本地 Tick 输入

- **目标**：建立无外部网络依赖的第一数据入口，避免 M2 被真实 Gateway 阻塞
- **依赖**：P1-T2、P1-T3
- **产出**：`SimGateway` 或等价本地回放输入模块；最小 Tick 样例数据；启动/停止/订阅接口
- **验证**：可稳定回放至少 1 个合约的 Tick 样例，并进入事件总线
- **阻塞条件**：输入接口与 `gateway_protocol.md` 或事件模型不一致

### P1-T5 Tick 持续落盘与 DuckDB 查询入口

- **目标**：建立最小本地数据飞轮起点
- **依赖**：P1-T4
- **产出**：TickCache、Parquet 落盘、DuckDB 查询入口、最小目录约定
- **验证**：回放 Tick 后可在本地查到同口径数据；读写路径可自动化测试
- **阻塞条件**：落盘数据结构与协议字段不一致，或查询结果不可复验

### P1-T6 RuleEngine 最小 session 判定

- **目标**：实现夜盘与 trading_date 最小规则
- **依赖**：P1-T2、P1-T4
- **产出**：交易时段模板加载、session 判定
- **验证**：夜盘样例测试通过；交易日归属与 session 切分可复验
- **阻塞条件**：规则样例与协议描述冲突

### P1-T7 BarAggregator + 1 个指标

- **目标**：完成 Tick -> Bar 与最小指标计算，并验证 Bar 归属不破坏交易日语义
- **依赖**：P1-T3、P1-T6
- **产出**：Bar 聚合器、至少 1 个指标
- **验证**：1 分钟 Bar 聚合测试通过；夜盘 Bar 的 session / trading_date 一致性测试通过
- **阻塞条件**：计算结果不稳定或不可复验

### P1-T8 UiBridge 最小投影

- **目标**：向前端提供最小 Query / Subscription 投影
- **依赖**：P1-T3、P1-T7
- **产出**：最小桥接接口与标准返回结构
- **验证**：契约测试与样例结构一致
- **阻塞条件**：桥接字段与协议样例偏离

### P1-T9 核心闭环演示

- **目标**：把前面任务串成可运行闭环
- **依赖**：P1-T1 ~ P1-T8
- **产出**：开发演示脚本 / README / 集成测试
- **验证**：SimGateway -> Tick 落盘 -> DuckDB 查询 -> Event -> Bar -> Indicator -> Snapshot 全链路可复现
- **进入下一阶段前提**：P1 gate 全部通过

## 4. P2：M3 插件系统任务包

### P2-T1 Manifest 与注册表

- **目标**：定义插件发现、注册、版本与依赖的最小治理面
- **依赖**：P1-T9
- **产出**：manifest schema、注册表结构、builtin 与第三方插件命名约定
- **验证**：合法/非法 manifest 样例可区分；依赖缺失可报明确错误

### P2-T2 PluginManager

- **目标**：实现插件加载、禁用、卸载与生命周期管理
- **依赖**：P2-T1
- **产出**：PluginManager、插件状态机、加载顺序控制
- **验证**：插件可发现、加载、禁用、卸载；重复加载与缺失插件有确定行为

### P2-T3 Host API 最小实现

- **目标**：提供只覆盖当前闭环的最小 Host API，而不是提前铺满全部能力
- **依赖**：P2-T2
- **产出**：读取市场数据、读取指标结果、输出信号、受控命令入口
- **验证**：插件可通过 Host API 完成一个完整的内建样板链路

### P2-T4 权限矩阵与隔离

- **目标**：实现 M3 默认隔离级别：线程级隔离 + 权限门禁 + 异常隔离
- **依赖**：P2-T2、P2-T3
- **产出**：权限检查、插件异常捕获、隔离执行边界
- **验证**：异常插件不会拖垮主流程；未授权调用被拒绝

### P2-T5 Builtin 插件样例

- **目标**：形成一条可以一键运行的内建插件样板，而不是先开放生态
- **依赖**：P2-T3、P2-T4
- **产出**：至少 2 个内建指标/信号插件、样板脚本、样板测试
- **验证**：在无外部网络条件下可运行完整插件闭环

### P2-T6 插件卸载 / 缺失恢复

- **目标**：保证工作区或运行链路在插件缺失时可降级恢复
- **依赖**：P2-T2、P2-T5
- **产出**：卸载策略、缺失插件恢复逻辑、降级说明
- **验证**：卸载或缺失插件后系统仍可进入稳定降级状态

每个任务包的统一验证要求：
- 插件加载路径可测试
- 未授权调用可拒绝
- 插件异常不拖垮主流程
- 工作区对缺失插件具备降级能力
- 内建样板可一键运行

## 5. P3：M4 回测系统任务包

### P3-T1 HistoricalDataLoader

- **目标**：优先消费本地 Parquet 落盘数据，避免再引入第二套早期数据口径
- **依赖**：P1-T5、P2-T5
- **产出**：HistoricalDataLoader、本地分片读取、最小筛选接口
- **验证**：可从本地落盘数据加载指定合约/时间段数据

### P3-T2 Replay Engine

- **目标**：建立与实时链路兼容的回放执行入口
- **依赖**：P3-T1
- **产出**：Replay Engine、回放控制接口、时间推进机制
- **验证**：回放数据可重放为标准事件流

### P3-T3 BacktestEngine

- **目标**：以共享协议对象与共享主链路逻辑为原则实现回测引擎
- **依赖**：P3-T2
- **产出**：BacktestEngine、最小策略执行循环
- **验证**：最小策略可运行，且不复制一套独立数据对象体系

### P3-T4 OrderMatcher

- **目标**：提供最小可解释撮合，不提前做过度复杂的市场微结构模拟
- **依赖**：P3-T3
- **产出**：限价单撮合、成交结果对象、最小费用/滑点钩子
- **验证**：撮合结果可通过手工样例复核

### P3-T5 Result / Report Builder

- **目标**：形成可复验的回测输出，而不是先追求复杂报告样式
- **依赖**：P3-T3、P3-T4
- **产出**：收益、回撤、胜率、交易明细与最小导出能力
- **验证**：输出结果稳定且可重复生成

### P3-T6 回放一致性测试

- **目标**：验证回放与实时链路共享同一口径，而不是只是“都能跑”
- **依赖**：P3-T1 ~ P3-T5
- **产出**：一致性测试集、黄金样例、差异定位输出
- **验证**：关键场景下一致性可自动化检查

统一验证要求：
- 回测策略样例可运行
- 输出结果可复验
- 与实时链路共享协议对象
- 与本地落盘 / 回放链路共享统一数据口径

## 6. P4：M5 产品壳任务包

### P4-T1 Tauri + React 初始化

- **目标**：建立桌面工作站壳，而不是只做静态前端原型
- **依赖**：P3-T6
- **产出**：Tauri 工程、React + TypeScript 初始化、基础运行脚本
- **验证**：可稳定拉起桌面壳并接入最小前端入口

### P4-T2 工作区壳与导航

- **目标**：形成日常工作流入口，包括数据、回放、风控、日志等基础导航位
- **依赖**：P4-T1
- **产出**：工作区壳、导航、最小状态组织
- **验证**：各核心页面入口可访问且状态切换稳定

### P4-T3 图表与面板基础布局

- **目标**：先建立可展示真实链路数据的图表与面板骨架
- **依赖**：P4-T2
- **产出**：Tick/K 线基础图表、指标区、日志区、状态区
- **验证**：可消费真实回放或实时链路数据，而不是仅静态 mock

### P4-T4 UiBridge 前端接入

- **目标**：把前端接到正式 Query / Command / Subscription 协议上
- **依赖**：P4-T1、P4-T3
- **产出**：桥接客户端、错误处理、订阅管理
- **验证**：前端可消费标准查询结果与标准事件

### P4-T5 Sidecar 生命周期与恢复能力

- **目标**：把 sidecar 的动态端口、健康检查、优雅关闭、恢复入口作为正式能力落地
- **依赖**：P4-T1、P4-T4
- **产出**：端口协商、健康状态检测、重启/恢复入口、最小诊断信息
- **验证**：核心进程异常退出、端口变化、恢复动作均可被前端感知并处理

### P4-T6 回放页与风控页基础交互

- **目标**：把回放和风控做成真实工作流入口，而不是仅用于展示
- **依赖**：P4-T3、P4-T4
- **产出**：回放控制、风控状态、最小操作反馈
- **验证**：用户可通过界面完成最小回放与风控查看路径

### P4-T7 打包与安装验证

- **目标**：在至少一个目标平台完成桌面打包与最小安装验证
- **依赖**：P4-T5、P4-T6
- **产出**：打包脚本、安装说明、已知限制
- **验证**：打包产物可启动；基本工作流可运行

统一验证要求：
- 可启动桌面壳
- 可消费标准查询与事件
- 最小演示路径稳定
- 用户可看到核心进程状态与恢复入口

## 9. 任务 AI 执行约束

交给任务 AI 执行时，必须遵守：

- 不得跨阶段抢跑
- 不得在 gate 未通过时开始下一阶段
- 不得用临时字段绕过协议
- 不得以“先写出来再整理”为理由污染边界
- 不得默认开启高风险 AI 能力
- 所有新增模块都必须附最小测试与验证命令

## 10. 任务包完成定义

一个任务包只有在以下条件全部满足时才算完成：

- 代码或文档产出已入仓
- 对应测试通过
- README / 契约 / 样例已同步（如适用）
- 风险与限制已记录
- 下一任务包所需输入已经稳定
