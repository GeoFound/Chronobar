# Chronobar 存储生命周期与恢复设计 v1.0

## 1. 文档定位

本文档定义 Chronobar 在本地桌面形态下的数据生命周期、目录约定、保留策略、备份/恢复路径、损坏检测与恢复原则。

它补充 [`architecture.md`](architecture.md)、[`config_protocol.md`](config_protocol.md) 与 [`../engineering/delivery_master_plan.md`](../engineering/delivery_master_plan.md) 中关于本地数据飞轮、用户主权与桌面恢复能力的约束。

## 2. 设计目标

1. 保证 Tick / Bar / 回放 / 配置 / 工作区等资产默认保留在用户本地
2. 保证数据路径可理解、可导出、可备份、可恢复
3. 保证局部损坏不会自动扩大为全局不可恢复故障
4. 保证回放与查询使用同一套规范数据口径

## 3. 存储分层

Chronobar 的本地存储分为三类：

### 3.1 时序数据层

用于保存 Tick、Bar、回放输入与事件日志。

- 主格式：`Parquet`
- 查询引擎：`DuckDB`
- 分片原则：按 `instrument_id / trading_date` 分片

### 3.2 元数据层

用于保存合约基础信息、交易日历、策略参数、索引信息与轻量注册表。

- 主格式：`SQLite`

### 3.3 配置与工作区层

用于保存系统配置、市场配置、规则配置、指标配置、工作区配置与用户模板。

- 主格式：`YAML` / `JSON`
- 校验：`JSON Schema` + 业务校验

## 4. 推荐目录约定

正式实现前，目录语义先冻结如下：

```text
<data_root>/
  ticks/
    <instrument_id>/
      <trading_date>.parquet
  bars/
    <instrument_id>/
      <interval>/
        <trading_date>.parquet
  replay/
    manifests/
    event_logs/
  metadata/
    app.db
  config/
    exported/
    backups/
  workspace/
    active/
    snapshots/
    backups/
  diagnostics/
    health/
    crash/
    recovery/
```

约束：

- 目录结构的稳定性优先于短期实现便利
- 不允许让 UI 临时缓存与正式工作区配置混存
- 不允许把导出包与运行中热写目录混为一体

## 5. 数据生命周期

### 5.1 Tick 数据

- 默认保存到本地 `ticks/`
- 按合约、交易日分片写入
- 允许后续归档，但归档后仍应保持可导入恢复

### 5.2 Bar 数据

- 允许作为派生数据单独缓存
- 若可从 Tick 重建，恢复优先级低于 Tick 主数据
- 当存储空间紧张时，优先清理可重建派生数据而非主 Tick 数据

### 5.3 配置与工作区

- 正式配置必须保留版本字段
- 工作区必须支持快照
- 导出配置时应分离业务配置与工作区布局

### 5.4 日志与诊断

- 运行日志允许按时间轮转
- 崩溃、恢复、数据损坏相关诊断信息应保留短期历史，便于用户手动导出

## 6. 保留与清理策略

正式实现前先冻结策略语义：

- **Tick 主数据默认不自动删除**，除非用户显式配置保留策略
- **清理优先级**：派生缓存 > 回放临时文件 > 旧诊断文件 > 用户明确允许清理的历史 Tick
- **自动清理动作必须可见**：任何归档、清理、压缩动作都必须有日志与用户可追溯记录

推荐的最小保留策略字段未来应支持：

- 按时间保留（例如保留最近 N 天）
- 按容量保留（例如磁盘超过阈值时清理低优先级数据）
- 按数据层保留（Tick / Bar / replay 临时文件分别治理）

## 7. 备份与导出

### 7.1 最小导出能力

必须支持以下导出路径：

- 指定时间段 / 合约的 Parquet 数据导出
- 配置导出
- 工作区快照导出
- 诊断包导出

### 7.2 备份原则

- 默认采用用户可理解的文件级备份，而不是隐藏式数据库快照
- 备份文件名应包含时间戳与版本信息
- 导出包必须包含 manifest，说明所含数据类型、时间范围、版本与生成时间

## 8. 恢复模型

### 8.1 配置恢复

当配置损坏或 schema 校验失败时：

1. 先尝试 migration
2. 再尝试最近一次可用备份
3. 再降级到默认配置
4. 保留损坏文件副本，供用户排查

### 8.2 工作区恢复

当工作区配置损坏时：

1. 尝试载入最近一次快照
2. 若失败，降级到默认工作区
3. 向用户显示“工作区已降级恢复”，而不是静默失败

### 8.3 数据恢复

当 Parquet 分片损坏时：

1. 标记为 `quarantine` 或等价隔离状态
2. 阻止其继续参与默认回放
3. 保留诊断信息
4. 允许用户从备份或导出包恢复

### 8.4 元数据恢复

当 SQLite 元数据损坏时：

1. 优先尝试从备份恢复
2. 若失败，允许重建非关键元数据索引
3. 不允许因为元数据损坏直接删除 Tick 主数据

## 9. 损坏检测

正式实现前先冻结以下检测方向：

- Parquet 文件不存在、无法读取或 footer 损坏
- manifest 缺失或版本不匹配
- SQLite 无法打开或 schema 不匹配
- 工作区文件缺少版本字段或包含非法字段
- 恢复链路中的 migration 失败

未来正式实现建议补充：

- 分片级 checksum
- 部分写入检测
- 启动时数据目录完整性扫描

## 10. 用户可见恢复入口

桌面路线要求恢复是产品能力，不是隐藏维护动作。

因此未来正式实现至少应提供：

- 查看当前数据目录状态
- 导出诊断包
- 重新加载默认配置
- 恢复最近一次工作区快照
- 重建可重建索引或派生缓存
- 明确的“重置到安全初始状态”入口

## 11. 阶段门禁

### 11.1 M1 / P0 必须完成

- 数据生命周期规则已成文
- 保留与清理原则已成文
- 配置 / 工作区 / 数据恢复模型已成文
- 用户可见恢复入口已被纳入正式范围

### 11.2 P1 / M2 必须验证

- Tick 主数据目录结构已有最小实现
- 至少 1 个导出 / 恢复路径可演示
- 数据损坏时不会导致整个系统静默崩溃

### 11.3 P4 / M5 必须验证

- 前端可感知核心健康与恢复状态
- 用户可从界面触发至少一种恢复动作

## 12. 禁止事项

- 禁止在未定义生命周期前直接堆积历史数据实现
- 禁止以“先跑起来”为由省略损坏处理与备份语义
- 禁止让工作区配置反向承担业务配置职责
- 禁止在恢复失败时静默重置或静默删除用户资产
