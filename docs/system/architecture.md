# 中国期货专用平台架构说明 v1.2

## 1. 文档定位

本文档定义平台整体架构、分层关系、模块边界、运行主流程和依赖约束。
它与数据协议、事件协议、配置协议、插件协议、工程基线、UI 桥接协议共同构成系统第一阶段的正式架构基线。

本文档描述的是平台长期稳定骨架，而不是某个具体页面或某个具体实现细节。
所有新增功能必须优先遵守本文档定义的边界，不得以临时跨层直连作为默认实现方式。

## 2. 目标用户

本平台面向两类核心用户：

1. **个人量化用户**：需要本地化、低延迟、可回测、可扩展的交易终端，支持自定义策略和指标。
2. **个人交易用户**：需要稳定易用的桌面终端，支持实时行情、技术分析和辅助决策。

两类用户对延迟、稳定性、合规性的要求相似：本地运行优先、数据隐私保护、回测结果可复验、策略安全可控。

## 3. 商业边界

本平台采用"开源核心 + 商业扩展"的双层模式：

- **开源核心**：协议层、架构层、基础引擎、插件框架、UI 桥接协议完全开源。
- **商业扩展**：高级策略库、专业数据服务、企业级风控、多账户管理等作为商业产品提供。

开源核心确保生态开放性和 AI 协作友好性，商业扩展确保可持续投入和专业化服务。

## 4. 架构目标

1. 建立一个适合中国期货市场的长期可扩展平台骨架。
2. 保证新增功能优先通过配置与插件外插，而不是改动核心内核。
3. 保证实时运行、历史回放、测试验证共享相同的数据协议与事件协议。
4. 保证 UI、网关、规则、计算、存储、插件各自独立演进。
5. 保证前端体验可持续升级，而不破坏核心交易链路与计算链路的稳定性。

### 4.1 平台骨架定位

Chronobar 的长期定位不是预设一套固定交易系统，而是提供期货交易所需的稳定骨架：

- 标准对象模型
- 事件协作通道
- 规则与风控边界
- 数据落盘与回放链路
- 网关适配规范
- UI Bridge
- 插件与 AI 协作边界

在这个骨架之上，用户可以通过配置、工作区、插件、策略、指标以及自己接入的 AI 能力，决定系统最终长成什么样。

平台负责保证边界稳定、语义一致、风险可控、行为可审计；平台不负责替用户预设唯一正确的策略结构、研究流程或界面形态。

### 4.2 用户资产模型

Chronobar 必须把以下对象视为用户资产，而不是临时运行副产物：

- 配置资产：系统、市场、规则、工作区及其迁移结果
- 数据资产：Tick、Bar、回放数据、导出物与诊断包
- 策略资产：策略代码、参数模板、信号模板与回测结果
- 指标资产：指标定义、可视化模板、解释说明
- 研究资产：因子定义、标签定义、实验配置、评估结果、研究报告与发布记录
- 连接资产：网关接入配置、能力声明、映射约定与恢复策略
- AI 资产：提示模板、候选产物、确认回执、受控记忆与研究记录

这些资产默认归用户本地持有，并应具备版本化、迁移、导出、恢复和审计能力。

### 4.3 构建态与运行态边界

Chronobar 必须区分构建态与运行态：

- 构建态：用户通过配置、插件、工作区、策略代码或 AI 候选产物来组装自己的系统
- 运行态：主引擎加载已确认、已校验、已发布的正式对象并稳定执行

边界要求：

- AI 主要在构建态提供解释、检索、候选产物与确认应用辅助
- 运行态不得把未确认候选产物直接视为正式对象
- 用户可以决定“系统做什么”，但不能破坏事件语义、风控路径、权限模型与审计边界这些稳定骨架

### 4.4 研究骨架与算法骨架

Chronobar 应支持个人用户在平台骨架上进行多因子研究、特征工程和策略装配，但不把某个默认算法预设为唯一正确路线。

平台至少应提供以下稳定骨架：

- 统一数据口径：Tick、Bar、session、trading_date、连续合约与换月语义
- 研究对象模型：因子定义、标签定义、实验配置、结果对象、发布记录
- 验证骨架：样本切分、walk-forward、成本模型、泄漏检查、稳健性检查
- 发布骨架：研究候选对象、正式对象、发布回执和回滚说明

算法策略上，平台负责提供最小算法接口与验证骨架；用户负责选择、配置、替换或扩展具体算法。

### 4.5 对象关系与资产图谱

Chronobar 的长期可维护性优先取决于对象模型，而不是页面数量。

平台至少应把以下对象关系视为正式结构，而不是分散在页面逻辑中的隐含约定：

- 因子定义依赖哪些原始数据字段、标签定义和实验配置
- 实验结果、研究报告、发布记录与正式运行态对象之间的来源关系
- 策略、指标、工作区、网关配置与已发布研究资产之间的引用关系
- AI 候选产物、确认回执、回滚说明与正式对象之间的变更关系

这些关系应逐步沉淀为 **Artifact Graph / User Asset Dependency Model**，用于影响分析、回滚、迁移、重写与审计解释。

### 4.6 状态机与三域边界

Chronobar 应长期坚持研究域、策略域、执行域三域分离：

- **研究域**：因子、标签、实验、评估结果、发布候选与解释对象
- **策略域**：策略逻辑、参数、风控约束、组合构建对象
- **执行域**：网关、订单、成交、持仓、账户与实盘状态

重要对象必须逐步拥有显式状态机，而不是依赖隐含流程语义。
例如研究对象可区分 `draft / validated / reproducible / pending_publish / published / deprecated / archived`；AI 候选对象可区分 `proposed / minimally_validated / pending_confirmation / applied / rejected / superseded / rolled_back`；网关适配可区分 `declared / verified / degraded / disabled`。

### 4.7 证据包、变更集与确认语义

AI 长期参与下，Chronobar 不应把高价值输出视为普通聊天文本，而应把以下对象逐步做成正式骨架：

- **Evidence Bundle**：来源对象、证据快照、推理约束、最小验证结果
- **AI Change Set**：待变更对象集合、依赖关系、风险等级、最小验证、回滚计划
- **Confirmation Receipt**：谁确认了什么、基于什么证据确认、是否可撤销

平台必须支持“可解释候选产物 -> 可确认变更 -> 可追溯正式对象”的闭环，而不是只保留最终结果。

### 4.8 能力注册表、策略数据化与国际化抽象

平台长期扩展不应依赖堆叠 if/else，而应优先沉淀以下正式抽象：

- **Capability Registry**：统一声明 gateway capability、market capability、AI capability、research engine capability、storage capability 与 compliance capability
- **Policy Engine as Data**：把 AI 权限、发布策略、双确认要求、外部访问策略、区域限制和默认拒绝规则写成数据与策略对象
- **Regional Market / Compliance Abstraction**：把交易日历、合约规范、订单能力、账户结算与监管约束抽象为可迁移、可本地化的正式层

目标不是在 M1 就实现所有国家/市场支持，而是在 M1 就预留稳定抽象位置。

### 4.9 迁移、重放与存储边界

Chronobar 必须把“未来可重写、可迁移、可跨国家平台移植”视为主骨架约束，而不是后补需求。

平台至少应逐步形成以下正式能力：

- **Migration / Portability Contract**：对象版本、Schema 迁移规则、审计记录迁移、外部引用保留与回放兼容策略
- **Deterministic Replay / Rebuild**：研究可重建、配置可重建、AI 候选可重建、回放结果可复核
- **Storage Boundary Contract**：区分 append-only audit store、config/object store、time-series market store、artifact graph store、semantic index 与 transient cache

只有在这些边界明确时，平台才真正具备长期维护、迁移和重写而不丢对象语义的能力。

## 5. 总体结构

平台采用七层架构：

1. 接口层
2. 事件层
3. 规则层
4. 计算层
5. 存储层
6. 展示层
7. 插件层

平台采用主引擎协调模式。
核心运行时由 Python 主进程承载，展示层采用 React 桌面前端，双方通过 UI Bridge 进行查询、命令和订阅协作。

## 6. 分层职责

### 6.1 接口层

职责：
- 适配 CTP / OpenCTP / SimNow 等底层接口。
- 接收原始行情、委托、成交、账户等回调。
- 将原始数据映射为 Tick、Order、Trade、Position、Account 等标准对象。
- 暴露网关能力声明，供上层判断当前接入支持的功能范围与降级路径。
- 向事件层发布标准事件。

约束：
- 不得直接调用展示层。
- 不得直接依赖前端状态。
- 不得绕过映射层传播网关私有字段。
- 不得把账户密钥、认证信息或私有连接细节泄露为跨层长期依赖。
- 原始字段如需保留，只能放入标准对象扩展区，不得形成跨层依赖。

### 6.2 事件层

职责：
- 提供统一事件总线。
- 管理事件注册、订阅、分发、定时器和链路追踪。
- 作为系统默认协作通道。
- 为实时模式与回放模式提供一致的事件承载模型。

约束：
- 事件分发必须线程安全。
- 单个 handler 异常不得阻塞总线。
- 必须支持全局订阅和 instrument 级精细订阅。
- 必须支持实时与回放共用协议。
- 所有可回放事件必须可序列化。

### 6.3 规则层

职责：
- 维护交易日历。
- 解析交易时段模板。
- 判定夜盘归属。
- 输出 SessionContext。
- 处理自然日、交易日、节假日、跨日时段和特殊品种规则。

约束：
- 不依赖具体前端实现。
- 不依赖某个具体 Gateway 的私有字段。
- 规则变化优先通过配置完成。
- 规则结果必须稳定、可测试、可回放。

### 6.4 计算层

职责：
- Tick -> Bar 聚合。
- 指标计算。
- 统计与信号生成。
- 因子计算、特征工程与标签生成。
- 研究实验执行与评估。
- 回放执行。
- 增量更新与全量重算。

约束：
- 只消费标准对象与规则结果。
- 不读取底层网关私有回调字段。
- 所有计算结果必须可序列化、可回放、可复验。
- 算法能力应通过稳定接口暴露，允许用户配置参数、模型与搜索空间。
- 不得为了前端展示需要而破坏计算语义。

### 6.5 存储层

职责：
- 保存配置。
- 保存元数据。
- 保存历史行情。
- 保存日志与回放结果。
- 保存因子定义、实验配置、评估结果与研究报告。
- 保存工作区、模板和用户布局。

约束：
- 不持有 UI 状态引用。
- 不直接参与业务规则判定。
- 不绕过事件系统直接驱动业务模块。
- 不把存储结构直接暴露给前端组件。

### 6.6 展示层

职责：
- 提供 React 桌面前端。
- 提供图表展示、副图展示、参数面板、工作区布局、日志与告警面板。
- 提供模板导入导出、联动控制、回放交互、主题切换和工作区编排。
- 通过 UI Bridge 获取初始化数据、执行命令并接收实时事件流。

约束：
- 只消费标准结果和插件输出。
- 不自行推导交易日和夜盘规则。
- 不直接依赖网关私有字段。
- 不将前端临时状态作为指标计算输入。
- 不得绕过 UI Bridge 直接访问核心模块。

### 6.7 插件层

职责：
- 提供指标、提醒、图层、导出器、规则扩展、信号扩展等能力。
- 通过 Host API 与平台交互。
- 复用事件总线参与系统协作。
- 支持实时模式与回放模式下的一致行为。

约束：
- 不直接修改核心对象内部状态。
- 不直接访问数据库连接。
- 不直接持有前端窗口对象或组件实例。
- 不得绕过权限模型执行越权操作。

## 7. 主引擎结构

平台采用主引擎协调模式，核心组件包括：

- MainEngine
- EventEngine
- GatewayManager
- RuleEngine
- ComputeEngine
- StorageManager
- PluginHost
- WorkspaceManager
- UiBridge

### 7.1 MainEngine 职责

- 启动和停止系统。
- 装配核心模块。
- 注册 Gateway。
- 注册插件系统。
- 装载配置。
- 对外提供统一服务入口。
- 装载已确认发布的用户资产，并拒绝直接加载未确认候选产物进入运行态。
- 协调展示层桥接服务。

### 7.2 EventEngine 职责

- 承接全部标准事件。
- 驱动模块解耦协作。
- 提供定时器事件。
- 为回放和排障保留 trace_id。
- 为前端订阅提供统一事件出口。

### 7.3 WorkspaceManager 职责

- 管理工作区定义、布局、模板、联动组、面板状态。
- 管理工作区保存、恢复、切换、导入导出。
- 向展示层提供标准化工作区视图模型。

### 7.4 UiBridge 职责

- 向 React 前端提供查询接口、命令接口和订阅接口。
- 管理标准对象到前端可消费结构的映射。
- 管理前端订阅、过滤、节流和批量推送策略。
- 保证展示层始终通过受控边界访问核心能力。

## 8. 运行主流程

### 8.1 实时模式

1. MainEngine 启动。
2. 读取系统配置与市场配置。
3. 初始化 EventEngine。
4. 初始化 RuleEngine。
5. 初始化 GatewayManager。
6. 初始化 ComputeEngine、StorageManager、PluginHost、WorkspaceManager。
7. 启动 UiBridge。
8. Gateway 建立连接并订阅行情。
9. Gateway 原始回调映射为 Tick。
10. EventEngine 发布 `TICK`。
11. RuleEngine 判定 session / trading_date。
12. ComputeEngine 聚合 Bar、计算指标。
13. PluginHost 分发给插件。
14. 存储层按策略写入日志、缓存与历史数据。
15. UiBridge 将标准结果推送给 React 展示层。
16. 展示层更新图表、面板、工作区和告警区域。

### 8.2 回放模式

1. MainEngine 切换到 replay 模式。
2. 从历史文件或数据库读取标准数据。
3. 按时间序列重建 EventEnvelope。
4. EventEngine 重新分发 `TICK / BAR / SESSION_*`。
5. 计算层与插件层按实时同协议运行。
6. UiBridge 将回放结果持续推送给展示层。
7. 展示层显示图表、时间轴、日志、信号和告警。
8. 回放结果可导出、可比对、可复验。

## 9. 依赖方向

允许依赖方向：

- 接口层 -> 事件层
- 规则层 -> 数据协议 / 配置协议
- 计算层 -> 数据协议 / 事件协议 / 规则结果
- 存储层 -> 数据协议 / 配置协议
- 展示层 -> UI Bridge / 标准结果 / 插件输出
- 插件层 -> Host API / 协议定义
- UiBridge -> MainEngine / EventEngine / WorkspaceManager

禁止依赖方向：

- 接口层 -> 展示层
- 规则层 -> UI
- 计算层 -> 前端组件
- 插件层 -> 核心对象私有状态
- 展示层 -> 网关私有字段
- 任意层 -> 绕过事件总线进行默认业务协作
- 展示层 -> 直接访问数据库或 Gateway 实例

## 10. 关键设计决策

### 10.1 为什么主语言选 Python

- 有利于快速建立事件驱动与插件式平台。
- 有利于规则表达、协议建模、回放验证和 AI 协同开发。
- 有利于先完成协议和边界，再逐步优化性能热点。

### 10.2 为什么展示层选 React

- 更适合复杂工作区、拖拽布局、图表交互与细粒度状态联动。
- 更适合构建一致的组件体系、主题体系和交互体验。
- 更利于未来扩展 Web 端与桌面端统一前端能力。
- 更利于前后端分层协作。

### 10.3 为什么内部核心边界不用 OpenAPI 直接替代协议文档

- OpenAPI 主要描述 HTTP 服务，不能替代本地对象、事件、规则和插件这类核心边界。
- 平台当前稳定边界仍然是数据协议、事件协议、配置协议和插件协议。
- 若后续引入远程配置服务、回放服务、告警服务，可再为远程接口单独补 OpenAPI。

### 10.4 部署形态决策

**最终部署形式：桌面客户端**

- **桌面容器**：Tauri
- **前端**：React
- **核心引擎**：Python sidecar 进程
- **通信方式**：本地 HTTP + WebSocket（UiBridge 唯一边界）
- **禁止**：React 直接调用 Gateway / Strategy / Repository
- **禁止**：将核心交易逻辑写入 Tauri Rust 层

**为什么选择 Tauri + Python Sidecar**

Tauri 本质上是用系统 WebView 渲染前端，而不是像 Electron 一样内置整套 Chromium，因此安装包和内存占用更小。对桌面交易工具来说，这很重要，因为还要同时运行 Python 策略、行情处理、回测和存储模块。

Python 主引擎作为 sidecar 进程运行，前端 React 做 UI，桌面壳负责拉起和管理它们。前端和内核职责非常清晰：React 只负责展示和交互，Python 只负责交易、回测、风控和数据处理。

### 10.5 存储层技术决策

**历史 Tick 存储：DuckDB + Parquet 双层架构**

- **实时写入路径**：GatewayAdapter → TickCache (内存 deque) → 定时落盘 → Parquet 文件（按交易日分片）
- **查询/回测路径**：BacktestEngine → DuckDB (直接查询 Parquet) → Strategy
- **Parquet 文件**：按 instrument_id/trading_date.parquet 分片存储，支持时间分区裁剪
- **DuckDB**：作为查询引擎直接读取 Parquet，不额外存储数据，仅作为 SQL 计算层使用

**元数据存储：SQLite**

- 合约基础信息、交易日历、策略参数等小表继续使用 SQLite
- DuckDB + Parquet 专门处理 Tick 时序数据

**为什么选择 DuckDB + Parquet**

- 零服务进程：DuckDB 是进程内库，符合单进程桌面应用架构
- AI 编写友好：DuckDB 的 Python API 极简，AI 生成的存储层代码出错率极低
- 回测一致性：回测与实盘使用完全相同的数据格式
- 性能优势：5M 行数据集下，DuckDB 写入 < 1秒、读取 < 1秒
