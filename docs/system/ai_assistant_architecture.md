# Chronobar AI Assistant 架构说明 v1.0

## 1. 文档定位

本文档定义 Chronobar AI Assistant 的产品级系统骨架，描述分层模型、运行链路、模块职责、依赖约束与演进边界。

它不替代 [`../core/ai_protocol.md`](../core/ai_protocol.md) 中对 AI 插件的协议定义，而是补充“Chronobar 如何把受控 AI 插件、检索能力、工具能力、治理能力和评测能力组织成一个可落地助手系统”。

本文档描述的是 M2 及之后的 AI Assistant 实施蓝图；M1 阶段只冻结边界和契约，不假定这些运行时模块已经全部实现。

## 2. 目标与非目标

### 2.1 目标

1. 让 AI 成为受控的交易助手，而不是自由执行的自治代理。
2. 让文档问答、回测解读、风控解释、配置解释共享统一编排骨架。
3. 让回答尽量建立在系统数据或正式文档证据之上，而不是纯生成。
4. 让风险边界、权限控制、来源保留、调用审计与评测成为一等能力。
5. 让后续 MCP 接入、多模型调度、混合工作流扩展有稳定演进路径。

### 2.2 非目标

- 不将 AI Assistant 设计为自动下单代理。
- 不允许 AI 绕过 [`ui_bridge_protocol.md`](ui_bridge_protocol.md) 或 Host API 直接访问核心私有对象。
- 不将模型输出视为事实真相源。
- 不在 M1 阶段引入大规模运行时实现。

## 3. 设计原则

- **受控优先**：AI 只能在明确边界内读取、解释、建议，不能越权执行。
- **证据优先**：优先引用系统数据、正式文档或受控外部来源。
- **路由优先**：先识别问题类型，再决定检索、工具调用、拒答或回答。
- **分级动作**：解释类动作可直接回答，修改类动作必须确认，执行类动作默认禁用。
- **可审计**：问题、路由、工具调用、来源、模型版本、回答结果必须可记录。
- **可评测**：产品问答、回测解读、风控解释、拒答等任务必须独立评估。

## 4. 九层模型

Chronobar AI Assistant 采用九层模型，用于把“聊天能力”提升为“受控交易助手系统”。

### 4.1 Client 层

职责：
- 提供聊天面板、侧边 Copilot、页内问答入口。
- 在回测页、风控页、配置页、帮助页内承载任务型交互。
- 展示来源、置信提示、拒答说明、确认按钮和操作回执。

约束：
- 不直接访问模型或底层工具。
- 所有请求必须通过受控桥接边界进入编排层。

### 4.2 State / Storage 层

职责：
- 保存短期会话状态。
- 保存当前页面、当前对象、最近问题、最近证据、最近工具调用结果。
- 为 AI Assistant 提供可清理、可过期的任务上下文。

约束：
- 默认只保留短期、受控、可删除状态。
- 不将长期记忆作为默认能力。

### 4.3 Orchestrator 层

职责：
- 识别请求类型。
- 决定走文档检索、结构化工具查询、拒答、澄清还是回答。
- 管理降级、重试、确认点和失败回退路径。

约束：
- 不能把所有问题无差别丢给 LLM。
- 必须先过策略检查，再进入工具或检索阶段。

### 4.4 Tool 层

职责：
- 对接系统内结构化能力。
- 支持查询回测结果、配置、工作区、风险事件、日志、订单、持仓等。

约束：
- 所有工具必须有稳定输入输出结构。
- 工具调用必须受权限矩阵和策略引擎限制。

### 4.5 Skill / Catalog 层

职责：
- 管理可发现能力单元。
- 为不同问题类型绑定 skill、prompt 模板、工具集合、适用页面和权限声明。

约束：
- skill 不是自由脚本入口，而是受注册表和策略控制的能力目录。

### 4.6 Semantic Index 层

职责：
- 为 README、协议文档、FAQ、帮助文档和错误说明提供语义检索。
- 将关键词搜索升级为证据检索。

约束：
- 仅索引正式受控知识源。
- 检索结果必须保留来源位置。

### 4.7 MCP / External Context 层

职责：
- 以标准化方式接入受控外部上下文。
- 支持交易所说明、规则库、内部知识服务、受控搜索等。

约束：
- 外部上下文默认关闭，按来源白名单接入。
- 必须明确标记为外部来源。

### 4.8 RAI / Governance 层

职责：
- 管理边界、权限、拒答、确认点、来源保留、调用链审计与模型使用规范。
- 保证 AI Assistant 满足金融敏感场景下的风险控制要求。

约束：
- 不提供投资建议。
- 不允许执行交易。
- 不允许无证据强答。

### 4.9 Evaluation 层

职责：
- 评估产品问答、回测解读、风控解释、配置解释和拒答表现。
- 跟踪正确率、引用率、误导率、拒答准确率等指标。

约束：
- 评测必须任务分层，不能只看单一主观分数。

## 5. 最小运行链路

Chronobar AI Assistant 的推荐最小运行链路如下：

```text
Client
-> Intent Router
-> Policy Engine
-> Retriever / Tool Executor
-> Evidence Binder
-> Answer Composer
-> Audit Logger
```

### 5.1 Intent Router

职责：
- 识别以下问题类型：
  - 产品问答
  - 回测解读
  - 风控解释
  - 配置解释
  - 概念说明
  - 范围外问题

输出：
- `intent_type`
- `confidence`
- `required_capabilities`
- `risk_level`

### 5.2 Policy Engine

职责：
- 判断请求是否允许回答。
- 判断是否允许调用某类工具或外部上下文。
- 判断是否需要二次确认。
- 判断是否必须拒答或要求澄清。

### 5.3 Retriever / Tool Executor

职责：
- 文档问题优先走知识检索。
- 数据问题优先走结构化工具查询。
- 混合问题允许“先工具、再检索、后解释”。

### 5.4 Evidence Binder

职责：
- 绑定文档证据、系统数据证据和外部证据。
- 形成可被 Answer Composer 使用的统一证据包。

### 5.5 Answer Composer

职责：
- 生成最终回答。
- 显示依据来源、动作级别、不确定性和下一步建议。

### 5.6 Audit Logger

职责：
- 记录问题、会话、路由、调用工具、证据来源、模型版本和回答结果。

## 6. Chronobar 最小能力面

M2 起建议优先落地以下三类高价值、低风险能力：

1. **产品问答 Copilot**
   - 回答协议、配置、错误码、功能说明。
2. **回测解读 Copilot**
   - 解释收益、回撤、胜率、滑点、异常区间。
3. **风控解释 Copilot**
   - 解释命中的规则、拦截原因和修复建议。

这些能力共用同一套编排骨架，但可绑定不同 skill、工具与策略配置。

## 7. 状态模型

AI Assistant 只应默认使用受控的短期会话状态，至少包括：

- 当前页面
- 当前工作区
- 当前合约
- 当前回测任务
- 最近一次问题
- 最近一次工具调用结果
- 最近一次证据包摘要

不建议在第一阶段引入跨会话长期记忆。

## 8. 工具能力边界

AI Assistant 工具分为以下几类：

- **知识工具**：查询正式文档、FAQ、错误码、帮助中心
- **系统查询工具**：查询回测结果、工作区、配置、插件、日志、风险事件
- **受控外部工具**：查询受白名单限制的外部文档或规则说明
- **修改类工具**：生成配置草稿、生成代码草稿、参数调整建议

约束：
- 修改类工具必须走确认流程。
- 执行型工具默认不纳入 AI Assistant 范围。

## 9. 与现有协议的关系

### 9.1 与 AI 插件协议的关系

- [`../core/ai_protocol.md`](../core/ai_protocol.md) 定义 AI 插件自身的接入与权限。
- 本文档定义 AI Assistant 作为“系统级助手”的编排结构。
- AI Assistant 可消费 AI 插件能力，但不被 AI 插件协议替代。

### 9.2 与 UI Bridge 协议的关系

- [`ui_bridge_protocol.md`](ui_bridge_protocol.md) 仍然是前端访问核心能力的唯一正式边界。
- AI Assistant 的 Client 层与 Tool 层必须通过受控桥接边界访问核心能力。

### 9.3 与系统架构文档的关系

- [`architecture.md`](architecture.md) 定义 Chronobar 主体七层架构。
- 本文档在其上补充 AI Assistant 的产品级辅助层与治理层，不改变主体依赖方向。

## 10. 依赖约束

允许依赖：

- Client -> Orchestrator
- Orchestrator -> Policy Engine / Catalog / Retriever / Tool Executor
- Retriever -> Semantic Index / 受控知识源
- Tool Executor -> UiBridge / Host API / 标准查询接口
- Answer Composer -> 证据包 / 模型接口
- Audit Logger -> 标准审计存储

禁止依赖：

- Client -> 核心模块私有对象
- LLM -> 直接访问数据库连接或 Gateway 实例
- 外部上下文 -> 绕过策略引擎直接进入回答阶段
- 修改类工具 -> 绕过确认流程直接执行
- 任意组件 -> 绕过审计直接调用高风险能力

## 11. 治理要求

Chronobar AI Assistant 至少必须满足以下治理要求：

- 明确声明不提供投资建议。
- 明确区分系统数据、正式文档、外部来源和模型推断。
- 对高风险或高不确定性回答显示不确定性提示。
- 对修改类动作要求明确确认。
- 对范围外问题、越权请求、缺少证据问题执行稳定拒答。
- 对所有关键路径保留审计记录。

## 12. 评测要求

至少建立以下任务级评测线：

- 产品问答正确率
- 回测解读一致率
- 风控解释准确率
- 配置解释正确率
- 来源引用命中率
- 拒答准确率
- 误导率

评测结果应服务于路由规则、提示模板、工具优先级和来源策略的持续改进。

## 13. 演进路线

### 13.1 Phase 1

- 产品问答 Copilot
- 回测解读 Copilot
- 风控解释 Copilot

### 13.2 Phase 2

- 概念解释
- 受控搜索
- Session State
- 来源展示与引用规范

### 13.3 Phase 3

- Skill Catalog
- Policy Center
- Evaluation Dashboard
- 研究 Copilot

### 13.4 Phase 4

- MCP Servers
- 多工具编排
- 多模型任务分配
- deterministic / dynamic / hybrid flow 并存
